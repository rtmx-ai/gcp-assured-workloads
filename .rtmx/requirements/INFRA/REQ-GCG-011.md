# REQ-GCG-011: Plugin Binary Release Workflow

## Overview

The plugin must produce platform-specific single binaries via an automated CI workflow triggered by git tags. aegis-cli discovers and invokes plugins as standalone executables -- the binary must work without Node.js installed on the target system.

## Specification

### Build Matrix

| Platform | Architecture | Binary Name |
|----------|-------------|-------------|
| macOS | arm64 (Apple Silicon) | gcp-assured-workloads-darwin-arm64 |
| macOS | x64 (Intel) | gcp-assured-workloads-darwin-x64 |
| Linux | x64 | gcp-assured-workloads-linux-x64 |
| Linux | arm64 | gcp-assured-workloads-linux-arm64 |
| Windows | x64 | gcp-assured-workloads-windows-x64.exe |

### Bundler

`bun build --compile` is the primary bundler. If bun cannot cross-compile for a target, use `pkg` or platform-specific CI runners.

### Release Trigger

- A git tag matching `v*` triggers the release workflow
- Full CI runs first (format, lint, typecheck, unit tests)
- If CI fails, no binaries are built or published

### Release Artifacts

Each release includes:
- Platform binaries (5 targets)
- `SHA256SUMS.txt` containing checksums for all binaries
- Source tarball (auto-generated by GitHub)
- Release notes (auto-generated from commits since last tag)

### npm Publish (Development Distribution)

In addition to binaries, the plugin is published to npm as `gcp-assured-workloads` for development use:
- Requires `@aegis/infra-sdk` as an npm dependency (not file: path)
- Plugin authors studying the exemplar can `npm install gcp-assured-workloads` and read the source

### Version Coordination

- Plugin version is independent of SDK version
- Plugin declares SDK compatibility via semver range: `"@aegis/infra-sdk": "^0.1.0"`
- Plugin version and git tag must match: tag `v0.2.0` requires package.json version `0.2.0`
- Plugin version is also embedded in the manifest output (verified by aegis-cli)

### Manifest from Binary (No Network Required)

The bundled binary must respond to `manifest` with zero network calls:
- The manifest is compiled into the binary as a static JSON object
- `./gcp-assured-workloads manifest` works on a machine with no internet

## BDD Scenarios

### Scenario 1: Tag push produces platform binaries
- Given a git tag `v0.2.0` is pushed
- When the release workflow runs
- Then 5 platform binaries are built
- And SHA256SUMS.txt is generated
- And a GitHub Release contains all artifacts

### Scenario 2: Binary works without Node.js
- Given the linux-x64 binary on a fresh Ubuntu container
- And Node.js is not installed
- When `./gcp-assured-workloads manifest` is executed
- Then valid JSON is emitted on stdout
- And exit code is 0

### Scenario 3: CI failure blocks release
- Given a git tag is pushed
- But lint fails
- When the release workflow runs
- Then no binaries are built or published

## Acceptance Criteria

- [AC1] 5 platform binaries produced per release
- [AC2] SHA256 checksums published alongside binaries
- [AC3] Binary responds to manifest without Node.js
- [AC4] Release blocked if CI fails
- [AC5] Tag-version match enforced
